# Klevby — план фундамента приложения

Этот файл — рабочий план, чтобы мы не прыгали между задачами, а последовательно вели фундамент приложения по этапам.

## Главный принцип

- Сначала архитектура и понятные владельцы поведения.
- Не добавлять параллельные дублирующие механизмы.
- Сначала понять, кто отвечает за поведение, потом менять.
- Приложение рассматриваем как будущий мобильный продукт для Google Play / Apple Store, а не просто сайт.

- PR2 — Auth Service: подготовка отдельного слоя авторизации без изменения поведения приложения.

## Текущий фокус

- Мы продолжаем блок “Фундамент приложения”.
- PR10 принят поверхностным тестом.
- Следующий фундаментальный шаг выбирается отдельно.
- Ленту пока не чиним, она остаётся в отложенных проблемах.

## Отложенные проблемы

- feed flicker / полная перезагрузка ленты при лайках/комментариях;
- возможная отдельная проблема загрузки фото в ленте;
- UX баннера обновления PWA слишком низко у нижней навигации;
- chat delete после resume работает, но может иметь задержку 3–4 секунды из-за fallback;
- общий resume/recovery всё ещё требует дальнейшей архитектурной чистки;
- installed PWA/cache могут давать ложные регрессии при тестах.

## Следующие шаги

- После создания этого файла продолжить фундамент, а не ленту.
- Следующий технический шаг должен быть выбран отдельно после проверки diff.
- Любые будущие PR должны явно указывать, какой слой/владелец поведения меняется.
- Не добавлять “ещё один fallback / ещё один refresh / ещё один listener” без объяснения, почему старый путь нельзя исправить или кто теперь владелец.

## Правила проверки PR

- Проверить Summary.
- Проверить diff.
- Убедиться, что изменён только `docs/klevby-foundation-plan.md`.
- Убедиться, что нет изменений поведения приложения.
- После merge сделать pull локально и проверить `git status --short`.
